Build Specification & Self-Assessment

TinyVM — a stack-based bytecode virtual machine

A stage-by-stage spec for building a bytecode VM and its assembler in C. It defines what each stage must achieve, how you'll know it's done, and the traps to watch for — but never the code. The implementation is yours.

§0 Ground rules — the assessment contract

  • No code or pseudocode is provided. Guidance is conceptual only: mechanisms, models, and the why. Translating to C is your job.
  • You must be able to explain every line you write. If you can't explain why something works, it isn't done — even if it compiles and runs.
  • Debugging help is allowed. When something won't compile or misbehaves, the reason is explained; the fix stays yours to type.
  • Build in layers. Each stage must run and be verified before the next. Add a capability when a program you're trying to run demands it — not speculatively.

◎ The end goal

A working VM that runs real programs written in a small assembly language, including recursive function calls. The headline acceptance test:

Acceptance: a recursive Fibonacci program, assembled from a text file, prints fib(20) = 6765.

If that works, the machine's core — stack, control flow, locals, and call frames — is provably correct.

△ The mental model — hold this throughout

Three independent pieces. Keeping them separate is what makes the project tractable and testable.

The program

An array of instructions (operation + optional operand). Static data — the recipe.

The VM

A loop with state: a program counter + an operand stack (later, a call stack). Reads instructions and performs them — the cook.

The assembler

A translator: text in, instruction array out. A convenience for humans. The VM never needs to know it exists.

Two regions of memory, two jobs: the program array (recipe — read top to bottom, never changes while running) versus the operand stack (countertop — where values actually move during execution).

Stage 0 Representation & disassembler Complete

Goal: prove the data model. No execution yet.

Must have

  • An enum of operations (the vocabulary).
  • An instruction struct: which operation + one operand value.
  • A hand-built program as an array of instructions.
  • A name helper: operation → name string. Single source of truth for names. (A switch returning string literals; no allocation.)
  • A disassembler loop that walks the array and prints each instruction in readable form, delegating naming to the helper.
Done when: the program prints its own instructions back in words (PUSH 10, DIVIDE, …), using a real loop whose length is computed from the array — not a hardcoded count.

Traps

  • Enum members are integers at runtime; names exist only in source. You must supply the integer→name mapping yourself.
  • Use name labels in switch cases (case PUSH:), never magic numbers (case 4:) — the latter breaks silently if the enum is reordered.
  • sizeof(array)/sizeof(array[0]) gives element count — but only where the real array is visible. It breaks the moment the array is passed to a function (decays to a pointer).
  • main returns int (status: 0 = success). Unrelated to any string return.

Stage 1 The execution loop

Goal: make it run. Instructions now do things instead of being printed.

New concepts

  • Operand stack — storage for values (an array) + a way to track the top. Decide: what holds the values, and what single piece of state tracks where the top is (updated on every push/pop)?
  • Program counter — an index for "which instruction is next."
  • Fetch–decode–execute loop — read the instruction at the counter, do what it says, advance. Structurally a switch over operations (the twin of your name helper, but each case acts instead of returning a name).

Must support (minimum to run 2 + 3)

  • A "push constant" operation.
  • Arithmetic: add, subtract, multiply, divide.
  • A way to observe a result (print top of stack).
  • A way to halt (stop the loop cleanly).
Done when: a hand-built program computes an arithmetic result and prints it correctly (e.g. 10 / 5, then + 3, prints 5).

Traps

  • In this switch, cases do not return — so each needs break. Forgetting it causes fall-through (one case bleeds into the next): a classic bug.
  • Pop order is reversed. The last value pushed is the first popped. Irrelevant for ADD/MUL, but for SUB/DIV you must decide which popped value is the left operand. Fix and document the convention now.
  • Guard the stack: pushing when full (overflow) or popping when empty (underflow) must fail loudly, not corrupt memory. Bounds-check early.
  • Divide-by-zero must be handled, not left to crash.

Stage 2 Control flow — jumps

Goal: branching and loops.

New operations

  • Unconditional jump (set the program counter to a target instruction).
  • Conditional jump(s) — e.g. jump if top-of-stack is zero / non-zero (which pops the tested value).
  • Comparisons (equal, less-than, …) that push a 1/0 result — same pop-pop-combine-push rhythm as arithmetic.
Done when: an iterative program using a loop works — e.g. sum 1..10 → 55.

Traps

  • A jump target is an index into the program array, not a byte offset (given fixed-size instruction structs). Be clear which.
  • Advance the program counter before executing, so a jump can cleanly overwrite it without being re-incremented.
  • Off-by-one loops: verify the boundary instruction runs the intended number of times.

Stage 3 Local variables

Goal: named storage beyond the operand stack.

New operations

  • Load a local (push the value in slot n onto the stack).
  • Store a local (pop the stack into slot n).

Concept: locals live in a fixed set of slots, addressed by number. For now, a single set of slots is enough (per-frame locals arrive in Stage 4).

Done when: an iterative factorial works using a local accumulator (e.g. fact(10) = 3628800).

Traps

  • Bounds-check the slot index.
  • Keep straight what's a stack value vs. a local value — different storage.

Stage 4 Function calls & frames ★ core stage

Goal: real function calls with their own local scope, enabling recursion. This is you re-implementing the call-stack mechanism you studied in systems.

New concepts

  • Call frame — bundles a return address + that call's own local slots.
  • Call stack — a stack of frames (separate from the operand stack).
  • Call operation — carries a target address and an argument count. It creates a new frame, moves the arguments off the operand stack into the new frame's locals, records where to resume, and jumps to the target.
  • Return operation — restores the program counter to the saved return address and discards the frame. The return value is already on the shared operand stack (the function left it there), so nothing extra is plumbed.

Decisions you must make and defend

  • Argument order: arguments are pushed left-to-right; popping is reverse — how do you place them into locals so their order is preserved?
  • Where does the return value live so the caller can use it seamlessly?
Done when: recursive Fibonacci prints fib(20) = 6765. Verify a second recursive function (e.g. GCD by Euclid) to prove it wasn't a fluke.

Traps

  • Each frame must have its own locals, or recursion clobbers itself.
  • Call-stack overflow (runaway recursion) must fail loudly.
  • Returning from the top-level frame is an error, not a jump into nowhere.
  • Key insight: the return value rides on the shared operand stack, so return moves almost nothing.

Stage 5 The assembler

Goal: replace hand-built C arrays with programs written in a text file (.tvm), so you can write real programs without recompiling the VM.

Concepts

  • Two-pass design. Pass 1: scan the text, assign each instruction an address, record every label: → its address. Pass 2: emit instructions, resolving label references (for jumps/calls) to addresses.
  • Why two passes: a jump or call may target a label defined later in the file (a forward reference). You can't resolve addresses until every label is known — hence a full first pass before emitting.
  • Parsing: strip comments, split into tokens, map mnemonics → operations, parse operands (integers, or labels to be resolved).
Done when: the Stage 4 Fibonacci program, now written as a .tvm text file and assembled, still prints 6765.

Traps

  • Program length is now unknown up front — this is the legitimate place for a growable buffer (the dynamic-allocation instinct finally belongs here).
  • Undefined labels, duplicate labels, and unknown mnemonics should be reported with a line number, not silently mishandled.
  • Keep the assembler entirely separate from the VM — the VM still only consumes an instruction array.

Stage 6 Polish — portfolio-grade

Goal: make it feel like a real tool. Cheap to add, disproportionately impressive.

  • A trace mode (flag) that prints each instruction as it executes, using the name helper — invaluable for debugging your own bytecode.
  • Robust runtime errors with context (over/underflow, div-by-zero, bad jump, bad slot) — fail loudly and clearly.
  • A small command-line interface: run a file, disassemble a file, run-with-trace. (Where argv finally gets used — with the correct char ** type.)
  • File organization: split into headers (declarations) and source (definitions) once VM and assembler are clearly distinct. Add include guards (they defend against the double-paste of #include).
  • A README: what it is, how to build/run, the instruction set, an annotated example, and the architecture (the three-piece model).
  • A few example programs (fib, factorial, gcd) checked in.

✓ Self-assessment rubric

Rate yourself honestly at each stage. "Done" means all three:

  1. It works — the stage's acceptance test passes.
  2. It's robust — bad input fails loudly (bounds, div-by-zero, bad targets), not with silent corruption or a mystery crash.
  3. You can explain it — every line, and why the design is the way it is.

A stage that passes (1) but not (3) is not complete for the purposes of this assessment — understanding is the deliverable; the binary is just evidence.
